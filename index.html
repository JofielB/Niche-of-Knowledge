The 4 Pillars of OOP: Today I learned that there are 4 principles in the OOP paradigm. I have already known them but I didn’t know that they were called that way. These 4 pillars are like recommendations of how to take advantage of the OOP paradigm:

  • Abstraction: This consists in obtaining the more particular or general attributes from an object and creating a class in base of them. Is very important to just abstract the more essential attributes of a class to make it possible to use in many situations.

  • Encapsulation: The encapsulation make reference to the liberty that other classes have to access to the attributes and methods of the other classes, for example if you are using a class called people you will be able to access to its public methods or attributes, for example let's said that the class has a attribute name and is public you will be able to use it but if the class has an attribute called id and is private your will not be able to use it.

  • Inheritance: This is the ability to inherit the methods and attributes from one class to another.

  • Polymorphism: This pillar is holding hands from the inheritance one because to have polymorphism is necessary to have inheritance. For example, if you have a class called Car with the basic attributes and methods that a car could have you can create a class called Bocho or Troca that could inherit the basic methods and attributes that a car has and implement them in its own way. In this way all the cars would have the same methods but implemented in different ways or forms.

Key points about the Mocks Aren't Stubs reading:

  • Mock Objects: These are special objects that mimic real objects for testing

  • Most languages have frameworks that make it easy to create mock objects.

  • The mock objects are one form to test objects

  • The xUnit test follows a typical four phases: sequence: setup, exercise, verify, and teardown.

  • jMock: Java library to make mock objects

  • SUT: System Under Test

  • Test Double: Generic term for any kind of pretend object used in place of a real object for testing purpose

  • Dummy objects are passed around but never actually used. Usually, they are just used to fill parameter lists.

  • Fake objects actually have working implementations, but usually take some shortcuts which makes them not suitable for production (an in-memory database is a good example).

  • Stubs provide canned answers to calls made during the test, usually not responding at all to anything outside what's programmed in for the test.

  • Spies are stubs that also record some information based on how they were called. One form of this might be an email service that records how many messages it was sent.

  • Mocks are what we are talking about here: objects pre-programmed with expectations which form a specification of the calls they are expected to receive.

  • More Info: https://martinfowler.com/articles/mocksArentStubs.html
